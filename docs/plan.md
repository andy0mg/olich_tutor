# Дорожная карта развития системы

## Организация работ

Документ фиксирует **этапы** развития продукта и платформы на высоком уровне; детальная декомпозиция ведётся в tasklist’ах областей по [docs/templates/workflow.md](templates/workflow.md). Связь с [docs/vision.md](vision.md): каждый этап даёт проверяемый инкремент, не перечень мелких задач. **Принцип:** один этап имеет **один ведущий tasklist** (область-лидер в таблице); кросс-компонентные этапы явно привязаны к этой области, смежные области подключаются через связанные задачи в своих tasklist’ах.

## Ключевые особенности плана

- **Ядро и API раньше второго клиента** — бизнес-логика и контракт backend остаются в центре; веб подключается к уже определённым сценариям и данным (см. компоненты в vision).
- **Инкремент персистентности** — сначала ценность на состоянии в памяти (MVP), затем БД по [ADR-001](adr/adr-001-database.md) и логической [data-model.md](data-model.md), без преждевременного усложнения схемы.
- **Один основной канал на этап** — сначала доводим Telegram-сценарии до проверяемого MVP, затем веб для ученика и родителя; параллельная работа — только где не блокирует стабильность контракта API (см. поля «Параллельность» у итераций).

## Итерации

### Итерация 1: Фундамент и Telegram-клиент

**Цель:** ученик получает ответы репетитора в Telegram с учётом истории сессии.

**⚡ Параллельность:** не требуется — базовый слой для последующих этапов.

**Критерии завершения (DoD):**

- Запуск по `make run`; сообщения проходят через слой тьютора/сессии; LLM-клиент stateless; логирование; секреты только из конфигурации.

**Связь с tasklist:** [docs/tasks/tasklist-bot.md](tasks/tasklist-bot.md)

**Полезный результат:** можно вести диалог с репетитором в Telegram (объяснения по запросу).

**Артефакты:** `main.py`, `bot/`, `tutor/session.py`, `llm/client.py`, конфиг — в духе структуры из [vision.md](vision.md).

---

### Итерация 2: MVP-учебные сценарии в боте

**Цель:** закрыть ключевые сценарии MVP из vision: запрос задания/теста, ответ ученика, обратная связь и фиксация результата и прогресса (пока в памяти).

**⚡ Параллельность:** после стабилизации границ API — подготовка контрактов для будущего веба (без обязательной реализации веба на этом этапе).

**Критерии завершения (DoD):**

- Сценарии воспроизводимы из бота end-to-end.
- Прогресс и результаты доступны в рамках сессии и зафиксированы в коде или краткой внутренней спецификации.
- Соответствие духу «ключевых сценариев MVP» в [vision.md](vision.md).

**Связь с tasklist:** [docs/tasks/tasklist-backend.md](tasks/tasklist-backend.md) (ведущая область; доработки бота — связанными задачами в [tasklist-bot.md](tasks/tasklist-bot.md), если области разведены).

**Полезный результат:** ученик не только получает теорию, но и проходит проверку с обратной связью.

**Артефакты:** расширение `backend/` / `tutor/` по фактической структуре репозитория; обработчики бота под новые шаги сценария.

**Текущий прогресс (фиксация по мере итерации):** REST API и сценарии MVP на стороне backend реализованы и описаны в [README.md](../README.md), [docs/vision.md](vision.md), OpenAPI — [`backend/openapi.yaml`](../backend/openapi.yaml); состояние — в памяти до этапа персистентности. Завершение **итерации 2** по критериям выше (в т.ч. сценарии из бота end-to-end через backend и инженерные задачи области) — по [docs/tasks/tasklist-backend.md](tasks/tasklist-backend.md), задачи 07–08.

---

### Итерация 3: Персистентность и модель данных

**Цель:** сохранение пользователей, диалогов и агрегатов прогресса между перезапусками; согласование с [data-model.md](data-model.md) и переносимыми миграциями.

**⚡ Параллельность:** не с полноценным вебом до готовности API поверх БД; описание контракта (например OpenAPI) может вестись параллельно финалу этапа.

**Критерии завершения (DoD):**

- Реализован выбранный в [ADR-001](adr/adr-001-database.md) путь (PostgreSQL / SQLite для dev); восстановление истории после рестарта.
- Нет критичной потери данных при штатном перезапуске сервиса; миграции переносимы между окружениями.

**Связь с tasklist:** [docs/tasks/tasklist-database.md](tasks/tasklist-database.md) (ведущий; задача 09 в [tasklist-backend.md](tasks/tasklist-backend.md) ссылается на него)

**Полезный результат:** прогресс и диалоги не обнуляются при обновлении или перезапуске сервиса.

**Артефакты:** слой данных и миграции; при необходимости согласованное уточнение [data-model.md](data-model.md).

---

### Итерация 4: Веб-клиент ученика и родителя

**Цель:** веб как второй клиент к тому же backend: сценарии ученика и просмотр активности и результатов родителем (см. [vision.md](vision.md) — веб).

**⚡ Параллельность:** с итерацией 5 по разным модулям — только при разделении команды и отсутствии конфликта по приоритету API; иначе этапы идут последовательно.

**Критерии завершения (DoD):**

- Согласованный MVP-способ входа: заготовка под OAuth из vision или явно зафиксированная альтернатива.
- Ключевые экраны опираются на реальные данные API, а не на заглушки.

**Связь с tasklist:** [docs/tasks/tasklist-web.md](tasks/tasklist-web.md)

**Полезный результат:** ученик и родитель видят прогресс и занятия не только через Telegram.

**Артефакты:** каталог `web/` (стек уточняется в рамках vision); интеграция с REST/backend.

### Итерация 5: Расширение платформы

**Цель:** роль преподавателя, материалы, потоки и группы — по продуктовому приоритету из [idea.md](idea.md), без противоречия [vision.md](vision.md).

**⚡ Параллельность:** с итерацией 4 — только после стабильного API и чёткого разграничения зон ответственности; иначе этап остаётся последовательным.

**Критерии завершения (DoD):**

- Выбранный инкремент (например, «преподаватель назначает материалы») работает end-to-end.
- Изменения отражены в tasklist/summary процесса по workflow.

**Связь с tasklist:** [docs/tasks/tasklist-backend.md](tasks/tasklist-backend.md); при необходимости правки [docs/tasks/tasklist-web.md](tasks/tasklist-web.md).

**Полезный результат:** операционное ведение учебного процесса, а не только диалог с ботом.

**Артефакты:** при значимых решениях — новые записи в [docs/adr/](adr/README.md).

---

### Итерация 6: Продакшн и сопровождение

**Цель:** надёжная эксплуатация: webhook вместо polling где целесообразно, деплой ([vision.md](vision.md) — деплой), резервное копирование, мониторинг логов.

**⚡ Параллельность:** с итерацией 5 — инфраструктурные задачи без изменения доменной модели, по согласованию с командой.

**Критерии завершения (DoD):**

- Задокументированный процесс деплоя; воспроизводимый перезапуск (systemd, Docker или эквивалент).
- Уровни логов согласованы с эксплуатацией; определён план резервного копирования БД.

**Связь с tasklist:** [docs/tasks/tasklist-infra.md](tasks/tasklist-infra.md)

**Полезный результат:** предсказуемая работа сервиса для пользователей и понятное обслуживание для команды.

**Артефакты:** скрипты или манифесты деплоя; при необходимости дополнения `.env.example`.
