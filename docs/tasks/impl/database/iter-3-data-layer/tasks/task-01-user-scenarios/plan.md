# Задача 01: сценарии пользователя и требования к данным — план

**Итерация:** iter-3-data-layer  
**Tasklist:** [docs/tasks/tasklist-database.md](../../../../../tasklist-database.md)

## Цель

Зафиксировать продуктовые сценарии **ученика** и **родителя**, вывести из них **требования к данным и связям** (в терминах [docs/data-model.md](../../../../../../data-model.md) и явных расширений для задачи 02), построить матрицу «сценарий → сущности/связи» с пометками **обязательно** / **позже** для персистентности итерации 3.

Опорные документы: [docs/idea.md](../../../../../../idea.md), [docs/vision.md](../../../../../../vision.md), [backend/openapi.yaml](../../../../../../../backend/openapi.yaml).

## 1. Сценарии ученика

Для каждого сценария: **цель**, **действие**, **что должно быть доступно после** (с точки зрения данных).

### 1.1 Диалог с AI-репетитором

| | |
|---|---|
| **Цель** | Получить объяснение темы, ответы на уточняющие вопросы, обратную связь в одной непрерывной беседе. |
| **Действие** | Ученик пишет в Telegram-бот (или позже в вебе); клиент создаёт/продолжает диалог, отправляет сообщения; backend хранит историю и вызывает LLM. |
| **Данные** | Идентификация ученика как субъекта обучения (`User` / `Student`); цепочка **`Conversation`** с каналом и временем; последовательность **`Message`** (роль, текст, порядок); опционально тема/контекст у диалога. |

### 1.2 Получение задания и проверка ответа

| | |
|---|---|
| **Цель** | Потренироваться на задачах и сразу понять ошибки. |
| **Действие** | Ученик просит задание или присылает решение; ответ формируется LLM; существенные реплики остаются в том же **`Conversation`** (как сообщения). |
| **Данные** | Те же **`Conversation`** / **`Message`**; фиксируются факт запроса задания, текст задания и ответов — для истории и для последующего анализа прогресса. Отдельная сущность «Задание» из [vision.md](../../../../../../vision.md) на этом этапе **не обязательна**, если упражнение представлено сообщениями. |

### 1.3 История занятий (диалогов)

| | |
|---|---|
| **Цель** | Вернуться к прошлым беседам, не терять контекст обучения. |
| **Действие** | Ученик открывает список диалогов и выбирает один (в вебе; в боте — упрощённый сценарий или последний диалог). |
| **Данные** | Список **`Conversation`**, отсортированный по активности; для выбранного — **`Message`** (с пагинацией). Привязка к **`Student`**. |

### 1.4 Прогресс по темам

| | |
|---|---|
| **Цель** | Видеть, что уже усвоено и где нужна доработка. |
| **Действие** | После занятия или по запросу фиксируется итог по теме; ученик смотрит сводку (веб / бот). |
| **Данные** | **`KnowledgeSnapshot`** по **`Student`** (тема, уровень, комментарий, время); опционально связь с **`Enrollment`** / **`LearningStream`** для отчёта «в рамках курса». **`LearningStream`** / **`Enrollment`** нужны, если прогресс разделяется по потокам. |

## 2. Сценарии родителя

### 2.1 Активность ребёнка

| | |
|---|---|
| **Цель** | Понимать, что ребёнок реально занимался: как часто, насколько недавно. |
| **Действие** | Родитель открывает веб (основной интерфейс для этой роли в [idea.md](../../../../../../idea.md)) и выбирает ребёнка из списка **привязанных** профилей. |
| **Данные** | Агрегаты по **`Conversation`** (последняя активность, количество/объём по периоду) и при необходимости по **`Message`**; всё относится к выбранному **`Student`**. Требуется **доверенная связь** родитель ↔ ребёнок (см. раздел 3). |

### 2.2 Результаты и динамика по темам

| | |
|---|---|
| **Цель** | Видеть усвоение по темам и «слабые места» без участия в переписке. |
| **Действие** | Родитель просматривает ленту/таблицу **`KnowledgeSnapshot`** выбранного ребёнка, фильтр по времени/теме. |
| **Данные** | **`KnowledgeSnapshot`**, опционально **`Enrollment`** / **`LearningStream`**; та же связь родитель ↔ **`Student`**. |

### 2.3 Несколько детей у одного родителя

| | |
|---|---|
| **Цель** | Вести учёт по каждому ребёнку отдельно. |
| **Действие** | Родитель переключает контекст (список детей → выбор одного); экраны 2.1–2.2 относятся к **текущему** выбранному **`Student`**. |
| **Данные** | Множество связей «родитель → ребёнок» (многие-ко-многим); в интерфейсе — только дети с **активной** подтверждённой связью. Сводка «все дети сразу» — **позже** (не обязательна для MVP персистентности). |

### 2.4 Несколько родителей у одного ребёнка

| | |
|---|---|
| **Цель** | Оба родителя (или опекуны) видят одну и ту же картину по ребёнку после принятия приглашения. |
| **Действие** | Каждый родитель проходит установление связи независимо; после подтверждения видит те же агрегаты по **`Student`**. |
| **Данные** | Та же связь M:N; **по умолчанию** — одинаковая видимость для всех подтверждённых родителей. Иная политика (разные права) — **позже**, если потребуется отдельное требование. |

## 3. Связь родитель–ребёнок: где нужна и как появляется

В [docs/data-model.md](../../../../../../data-model.md) нет отдельной сущности связи; для сценариев родителя она **обязательна**. Требование: связь **многие-ко-многим** между учёткой родителя и профилем ученика (**`User`** с ролью родителя ↔ **`Student`**), с жизненным циклом **приглашение → подтверждение** (или отклонение).

### 3.1 Где связь обязательна

- Доступ к данным ребёнка **только** при наличии доверенной связи в статусе «активна»; до этого — данные недоступны или отображается только состояние «ожидает» / «нет доступа».
- Список детей в кабинете родителя строится по **подтверждённым** связям.
- Подтверждение второй стороной обязательно: «висящее» приглашение не даёт полного доступа к учебным данным ребёнка.

### 3.2 Сценарии приглашения и подтверждения (продуктовый уровень)

**Цепочка:**

1. **Инициация** — один из вариантов (конкретный UX — на усмотрение следующих итераций; для требований к данным достаточно перечислить события):
   - родитель запрашивает привязку к ребёнку по **коду/ссылке**, сгенерированному у ребёнка (или у уже привязанного взрослого);
   - или ребёнок инициирует приглашение родителя по контакту/идентификатору.
2. **Доставка** — уведомление или экран в боте/вебе у второй стороны.
3. **Подтверждение** — вторая сторона принимает; связь переходит в активную.
4. **Отклонение** — связь не создаётся или помечается как отклонённая.
5. **Отзыв** — уже активная связь может быть снята инициатором или с политикой «выход» (детали — позже).
6. **Истечение** — приглашение с ограниченным сроком переходит в просроченное (опционально для MVP данных).

### 3.3 Статусы связи (требование к модели, детали таблиц — задача 02)

Минимальный набор для логики доступа:

| Статус (логический) | Смысл |
|---------------------|--------|
| Ожидает подтверждения | Приглашение создано, вторая сторона не ответила. |
| Активна | Доступ к данным ребёнка для этого родителя разрешён. |
| Отклонена | Явный отказ; повторное приглашение — новая попытка. |
| Отозвана | Связь снята после того, как была или могла быть активной. |

Дополнительно для удобства: **истекло** (если вводится срок действия приглашения) — может маппиться на отдельный флаг или на очистку записи.

### 3.4 Требования к сущностям (вход для задачи 02)

- Нужна возможность хранить **M:N** «родительский пользователь ↔ `Student`» с полем **статуса** и метками времени (создано, обновлено, опционально истекает).
- Для приглашения по коду/ссылке обычно нужны **непрозрачный токен** и **срок действия** — это может потребовать **отдельной сущности «приглашение»** или полей в той же таблице связи; **решение по нормализации и именам таблиц** принимает задача 02.
- Роли: в [data-model.md](../../../../../../data-model.md) у **`User`** указаны роли; для родителя нужна идентификация **`User`**, который выступает как родитель (без дублирования **`Student`** в одном лице с родителем — это разные роли одной учётки или разные учётки).

**Пробел относительно текущего data-model:** связь родитель–ребёнок и при необходимости сущность приглашения — **новые элементы логической модели** для включения в [docs/data-model.md](../../../../../../data-model.md) в **задаче 02**.

## 4. Матрица «сценарий → сущности и связи»

**Условные обозначения:** **M** — обязательно для персистентности итерации 3 / MVP данных; **P** — позже или необязательно для первой схемы; **—** — не используется.

Сущности и связи из [data-model.md](../../../../../../data-model.md): `User`, `Student`, `LearningStream`, `Enrollment`, `Conversation`, `Message`, `KnowledgeSnapshot`. Дополнительно: **Link** — связь родитель (`User`) ↔ `Student` (M:N, статусы приглашения); **Invite** — данные приглашения (токен, срок), если выносится отдельно (решение в задаче 02).

| Сценарий | User | Student | LearningStream | Enrollment | Conversation | Message | KnowledgeSnapshot | Link (родитель–ребёнок) | Invite (опц.) |
|----------|:----:|:-------:|:--------------:|:----------:|:------------:|:-------:|:-----------------:|:----------------------:|:---------------:|
| 1.1 Диалог | M | M | P | P | M | M | — | — | — |
| 1.2 Задание / проверка | M | M | P | P | M | M | P | — | — |
| 1.3 История диалогов | M | M | — | P | M | M | — | — | — |
| 1.4 Прогресс по темам | M | M | P | P | — | — | M | — | — |
| 2.1 Активность ребёнка | M | M | P | P | M | P | — | M | P |
| 2.2 Динамика по темам | M | M | P | P | — | — | M | M | — |
| 2.3 Несколько детей | M | M | — | — | — | — | — | M | P |
| 2.4 Несколько родителей | M | M | — | — | — | — | — | M | P |
| 3.2–3.3 Приглашение / статусы | M | M | — | — | — | — | — | M | P |

**Пояснения:**

- **LearningStream / Enrollment:** для MVP данных достаточно одного потока по умолчанию; полная опора на потоки — **P**, пока сценарии не требуют явного разделения курсов.
- **Link:** без активной связи сценарии 2.x для родителя невозможны; **Invite** помечен **P**, если токен и срок хранятся внутри той же записи связи без отдельной таблицы.
- Сценарии **родителя** не требуют прямого чтения **`Message`** родителем в первой версии продукта; достаточно агрегатов по **`Conversation`** — поэтому для 2.1 **Message** = **P** (детальный просмотр переписки родителем — отдельное продуктовое решение).

## 5. Сопоставление с текущим HTTP API (OpenAPI v1)

Источник: [backend/openapi.yaml](../../../../../../../backend/openapi.yaml), префикс `/api/v1`, идентификация через `X-Channel` и `X-External-User-Id`.

| Сценарий (ученик) | Покрытие API v1 |
|-------------------|-----------------|
| 1.1–1.2 Диалог, задание | `POST /conversations`, `POST /conversations/{id}/messages`, при необходимости `GET /conversations/{id}` |
| 1.3 История | `GET /conversations/{id}` (список диалогов в спецификации **не описан** — появится при расширении API или отдельном ресурсе) |
| 1.4 Прогресс по темам (фиксация) | `POST /knowledge-snapshots` |

| Сценарий (родитель) | Покрытие API v1 |
|---------------------|-----------------|
| 2.1–2.4 | **Не покрыты:** нет эндпоинтов для связи родитель–ребёнок, списка детей, агрегатов активности и выборки снимков «для родителя». Идентификация только «внешний пользователь канала» — для родителя в вебе потребуется отдельная модель сессии/учётки и авторизация (см. [vision.md](../../../../../../vision.md), OAuth — перспектива). |

Вывод: текущий контракт достаточен для **ученика** и сценариев, уже реализованных в итерации 2; **родительские** сценарии и **приглашения** потребуют **расширения API и модели пользователя** в следующих задачах backend/веб — это не входит в объём задачи 01, но зафиксировано для ожиданий.
