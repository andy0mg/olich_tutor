# Задача 01: сценарии пользователя и требования к данным — summary

**Итерация:** iter-3-data-layer  
**Tasklist:** [docs/tasks/tasklist-database.md](../../../../../tasklist-database.md)  
**План:** [plan.md](./plan.md)

## Что сделано

- Описаны сценарии **ученика**: диалог с репетитором, задание/проверка ответа, история диалогов, прогресс по темам — с привязкой к сущностям из [docs/data-model.md](../../../../../../data-model.md).
- Описаны сценарии **родителя**: активность ребёнка, динамика по темам, учёт **нескольких детей** у родителя и **нескольких родителей** у ребёнка; зафиксирована по умолчанию **одинаковая видимость** для подтверждённых родителей.
- Выделен раздел **связь родитель–ребёнок**: обязательность M:N, статусы (ожидает / активна / отклонена / отозвана), цепочка приглашения, требования к данным (в т.ч. опциональная отдельная сущность приглашения с токеном и сроком — решение в задаче 02).
- Построена **матрица** сценарий → сущности/связи с пометками **M** (обязательно для MVP данных) и **P** (позже).
- Добавлено **сопоставление с OpenAPI v1**: что покрыто для ученика, что отсутствует для родителя и приглашений.

## Решения и допущения

- Сводка по всем детям родителя одновременно и детальный просмотр переписки родителем отложены (**P**).
- Разная видимость для двух родителей не закладывается в обязательные требования (паритет после подтверждения).
- Конкретный UX приглашения (код vs ссылка, кто инициирует) оставлен на следующие итерации; для данных перечислены необходимые **события и статусы**.

## Передача в задачу 02

- Включить в логическую и физическую модель: **связь родитель (`User`) ↔ `Student` (M:N)** со статусами и аудитом; при необходимости — **приглашение** (токен, срок).
- Уточнить [docs/data-model.md](../../../../../../data-model.md) и ER после принятия решения по таблицам.
