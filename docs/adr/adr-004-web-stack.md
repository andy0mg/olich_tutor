# ADR-004: Стек веб-клиента (React + Vite + TypeScript)

| Поле | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-04-06 |
| Контекст | [vision.md](../vision.md), [plan.md](../plan.md) (итерация 4) |

## Контекст

Итерация 4 предусматривает веб-клиент для ученика и родителя, подключаемый к существующему REST API backend. В [vision.md](../vision.md) указано: «лёгкий SPA (React или аналог) с минимальными зависимостями». Веб — второй клиент к единому backend наряду с Telegram-ботом.

Требования: SPA с клиентским роутингом, поддержка двух ролей (ученик, родитель), интеграция с JSON REST API, современный UI без тяжёлых компонентных библиотек.

## Решение

1. **React 19 + TypeScript** — UI-библиотека и типизация.
2. **Vite** — сборщик и dev-сервер (быстрый HMR, минимальная конфигурация).
3. **Tailwind CSS** — utility-first стилизация.
4. **TanStack Query (React Query)** — кеширование и синхронизация данных с API.
5. **React Router v7** — клиентский роутинг с защищёнными маршрутами.

Каталог: `web/` в корне монорепо. Менеджер пакетов: npm.

## Обоснование

- **React** — крупнейшая экосистема, наибольшая база знаний; отвечает рекомендации vision.
- **Vite** — де-факто стандарт для новых React-проектов; быстрее CRA/webpack, нулевая конфигурация для базового SPA.
- **TypeScript** — типизация API-контрактов снижает ошибки интеграции frontend/backend.
- **Tailwind CSS** — быстрая стилизация без привязки к UI-фреймворку; лёгкий бандл благодаря tree-shaking.
- **TanStack Query** — декларативное управление серверным состоянием, retry, кеширование, оптимистичные обновления.

## Рассмотренные альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрано |
|---|---|---|---|
| Vue 3 + Vite | Меньше шаблонного кода | Меньше экосистема; vision упоминает React | Нет преимущества для данного проекта |
| Next.js | SSR/SSG, file-based routing | Избыточная сложность для SPA-клиента к API; отдельный Node-сервер | Противоречит «лёгкий SPA» |
| Svelte / SvelteKit | Компактный бандл | Меньше экосистема и специалистов | Выше риск для сопровождения |

## Последствия

- Появляется `web/` с собственным `package.json`; сборка и dev-сервер — отдельно от Python-процессов.
- Makefile пополняется целями `web-install`, `web-dev`, `web-build`.
- Деплой: статика из `web/dist/` раздаётся nginx или backend (при необходимости); детали — в итерации 6.
