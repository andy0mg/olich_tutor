# ADR-002: REST-слой backend (FastAPI + Uvicorn)

| Поле | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-04-05 |

## Контекст

Нужен единый HTTP API для клиентов (Telegram-бот, затем веб): сценарии учебного сопровождения, валидация входных данных, описание контракта для второго клиента без расхождения с реализацией. Стек уже зафиксирован: Python 3.12+, `pydantic-settings`, асинхронный бот на `aiogram`.

## Решение

1. **Фреймворк REST API — FastAPI.** Схемы запросов/ответов на Pydantic v2, автоматическая генерация OpenAPI, совместимость с async-обработчиками.

2. **Сервер приложения — Uvicorn** с extras **`[standard]`** (uvloop/httptools где доступны) для локального запуска и типового деплоя ASGI.

3. Контракт API для внешних команд — **OpenAPI** как источник правды наряду с кодом (детали — в задачах итерации backend).

## Обоснование

- **FastAPI** снижает объём шаблонного кода для API, даёт единый стиль с остальным стеком (Pydantic) и ускоряет согласование контрактов с фронтом и ботом.
- **Uvicorn** — де-факто стандарт для ASGI-приложений на Python; `standard` покрывает типичные нужды производительности без привязки к облачному вендору.

## Рассмотренные альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрано |
|---|---|---|---|
| Starlette «голый» | Минимум зависимостей | Больше ручной работы со схемами и документацией API | Дублирование того, что даёт FastAPI поверх Starlette |
| Flask / синхронный стек | Знакомство команды | Нет нативного async под текущий бот и I/O к LLM | Несоответствие направлению vision |

## Последствия

- Зависимости: `fastapi`, `uvicorn[standard]` в `requirements.txt`; запуск backend — отдельный процесс от бота.
- При смене фреймворка — новый ADR; OpenAPI остаётся обязательным артефактом контракта для клиентов.
